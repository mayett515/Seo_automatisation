# 2026-06-29 Progress

## Scope

Backend foundation hardening after architecture/pattern-mining reviews, ending with deploy retry hardening and the Netlify artifact-handoff baseline.

## Completed

- Added per-project publishable tracking keys with hashed storage, origin allowlists, create/list/revoke management, and public `/track` ingestion.
- Hardened `/track` with IP, IP/project, true project, key, and key/project rate-limit buckets.
- Coalesced tracking-key `lastUsedAt` writes so accepted events do not update the key row on every request.
- Replaced synthetic release preflight with persisted evidence-backed checks for approval, noindex, local SEO blockers/warnings, rollback readiness, and tracking readiness.
- Made release plan creation transactional and reject missing/cross-project page versions instead of silently dropping them.
- Made deploy approval transactional and state-machine guarded.
- Made deploy execution require a persisted approval row in addition to passing preflight checks.
- Split worker routing from GSC sync implementation and kept `job_runs` lifecycle updates in shared worker helpers.
- Moved `queueNames` to `@localseo/contracts` so API and worker share queue-name truth.
- Added `deployment_key`, deployment evidence JSON, and `release_verification_checks` schema support.
- Made migration `0009_lying_namor` backfill-safe by adding `deployment_key` nullable, filling existing rows, then enforcing `NOT NULL`.
- Expanded `SiteHostingPort` to provider-neutral create/read/restore/rollback methods.
- Changed `NotConfiguredSiteHostingAdapter` to return `not_configured` instead of fake deploy success.
- Implemented the deterministic deploy reconciler worker baseline:
  - parses shared `DeployJobDataSchema`,
  - derives a stable `deploymentKey`,
  - reloads release plan, checks, approval, rollback, and item evidence,
  - writes or reuses a deployment ledger row by `deployment_key`,
  - calls `SiteHostingPort.createDeploy` and treats `ready` as provider-ready/live, not merely accepted/building,
  - marks deployment/release as `provider_succeeded`/`live` on provider success,
  - keeps transient provider failures and pending provider states retryable until the final BullMQ attempt,
  - marks deployment/release as `failed` on terminal provider/configuration/evidence failure,
  - replays already-successful deployment rows without calling the provider again,
  - prevents retry startup from downgrading already-successful deployment ledger rows.
- Hardened deploy retry/audit behavior from the follow-up reviews:
  - terminal deploy configuration/evidence errors are rethrown to BullMQ as unrecoverable failures,
  - final-attempt detection prefers the producer-provided `maxAttempts` payload while retaining BullMQ option fallback,
  - failed deploys now insert-or-update the deployment ledger row by `deployment_key`, even when evidence gating fails before provider mutation,
  - duplicate stable BullMQ job IDs are backed by a DB unique key on `job_runs.external_job_id + queue_name`, so duplicate enqueue races reuse one audit row.
- Wired the approved artifact and Netlify hosting baseline:
  - added `ApprovedReleaseArtifactSchema`,
  - added filesystem `ObjectStoragePort` storage for local/test and S3-backed object storage for production,
  - the deploy worker writes approved release items/page-version JSON to `releases/{releasePlanId}/approved-artifact.json`,
  - `SiteHostingPort.createDeploy` can now return `pending` with a provider deploy id,
  - added a Netlify adapter using the official SHA1 digest deploy flow and mapping only Netlify `ready` to provider success,
  - the worker persists pending provider deploy ids and leaves them for reconciliation instead of overclaiming success.
- Extracted approved-release static HTML rendering into `@localseo/domain` so the Netlify adapter only handles provider upload mechanics.
- Added `provider_operation_status` to the deployment ledger and mark provider mutations `in_flight` before Netlify create calls. If a retry sees an in-flight provider mutation with no recorded provider deploy id, it fails closed for manual reconciliation instead of creating a duplicate provider deploy.
- Hardened the first productive hosting follow-up:
  - first deploys no longer require a rollback point because there is no prior live deployment to snapshot; later deploys still require rollback evidence,
  - deploy jobs now use a longer fixed retry budget and terminal Redis jobs no longer suppress legitimate re-enqueue,
  - terminal `job_runs` rows are archived before a legitimate re-run so historical audit rows are not overwritten,
  - pending provider deploys stay reconcilable instead of being marked failed only because the BullMQ retry window ended,
  - the worker now periodically reconciles persisted `deploying` rows that already have a `providerDeployId`,
  - the Netlify adapter polls async digest deploys until required file digests are available, uploads required files as `application/octet-stream`, and fails loudly if Netlify asks for a digest outside the approved artifact.
- Updated `docs/architecture/backend-foundation-status.md` so it reflects the new deploy reconciler baseline and the remaining provider/verification work honestly.
