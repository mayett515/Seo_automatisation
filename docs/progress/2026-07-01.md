# 2026-07-01 Progress

## Safety Hardening After Rollback Baseline Merge

- Merged the rollback lifecycle baseline into `main` through PR #1.
- Started the follow-up safety hardening slice on `codex/verification-redirect-safety`.
- Hardened the HTTP verifier redirect model:
  - live page fetches now follow redirects manually,
  - sitemap fetches now follow redirects manually,
  - redirect hops must stay on the original deployment origin,
  - off-origin redirect attempts become verification evidence instead of being fetched.
- Added adapter tests proving:
  - same-origin redirects still work,
  - live page redirects to off-origin/internal URLs are blocked,
  - sitemap redirects to off-origin/internal URLs are blocked.
- Updated foundation docs, ADR 0011, lifecycle hardening notes, and integration coverage docs so rollback point preparation is no longer described as a future gap and source selection is recorded as verified-good first with provider-succeeded fallback.
- Merged the redirect hardening slice through PR #2 and started `codex/rollback-point-idempotency`.
- Added a rollback point source identity guard:
  - the database now has a unique index on `(release_plan_id, deployment_id, provider_deploy_id)`,
  - release preflight uses conflict-safe insertion for prepared rollback points,
  - the migration deduplicates exact historical source duplicates before creating the index,
  - integration coverage proves duplicate source insertion is conflict-suppressed.
- Started `codex/deploy-lifecycle-status-alignment` after the post-merge architecture reviews found one concrete lifecycle mismatch:
  - deploy worker rollback-evidence checks now use the same safe prior-deployment status set as API preflight,
  - stale deploy replays for `rollback_recommended`, `verifying`, or `rolled_back` deployments no longer project the release back to `live`,
  - unit coverage proves `rollback_recommended` replay is a no-op for live projection,
  - worker integration coverage proves unsafe prior deployments do not force rollback evidence the API cannot safely prepare.
- Started `codex/provider-production-hardening` for the GPT production-readiness findings:
  - Netlify and Google Search Console adapter failures now use safe typed provider errors instead of persisting raw response bodies through worker audit paths,
  - Netlify and Google Search Console HTTP calls now have adapter-level timeout boundaries,
  - GSC sync now persists connection-level failure state for decrypt, invalid-refresh-token, transient refresh, and query failures,
  - reconnect-required GSC failures mark the connection `error`; transient/query failures leave it `connected` but visible through `failureJson`.
- Recorded the repo-inspiration run as a source note and accepted ADR 0012 as the production-readiness policy batch:
  - production accepted-event tracking should fail closed when Redis-backed write-protection limits are unavailable, while pre-validation may use only a soft local throttle that does not prove persisted acceptance,
  - rollback remains manual for MVP until pending rollback reconciliation and policy gates exist,
  - pending rollback reconciliation should observe provider state instead of re-posting restore mutations,
  - reconnect-required GSC failures should stay terminal and precise through retries,
  - provider-backed deploy timeout/unknown outcomes should stay reconcilable instead of becoming ordinary final-attempt failures.
- Started `codex/production-policy-guards` to implement the non-reconciler ADR 0012 guards before designing pending rollback reconciliation:
  - reconnect-required GSC sync failures now map to terminal worker errors and BullMQ unrecoverable rethrows,
  - accepted-event tracking rate limits now fail closed under strict/production Redis outages while pre-validation remains a soft local throttle,
  - provider-backed deploy upload/read uncertainty stays reconcilable after final attempts and in the deploy reconciler; explicit provider terminal failure now marks failed immediately and stops retries.
- Recorded the pattern-mining GSC OAuth boundary cleanup in the hardening backlog:
  - when next touching GSC OAuth, `SearchConsolePort` should own the full `createAuthorizationRequest(...)` operation so the API depends on the purpose port rather than `GoogleSearchConsoleAdapter`,
  - PKCE/state signing stays in the adapter, while Redis state persistence stays in the API-owned `GscOAuthStateStore`.
- Accepted ADR 0013 for the pending rollback reconciler storage direction:
  - canonical rollback operation vocabulary is now pinned before table implementation,
  - current JSON evidence should use `restore_in_flight`, `rollback_pending`, `completed`, `provider_failed`, and `manual_reconciliation_required`,
  - `superseded` is reserved for a future `rollback_operations` table, while `stale_noop` remains a function/metric outcome,
  - job retry owns `restore_in_flight` and final attempts must not leave it orphaned.
- Started `codex/pending-rollback-reconciler` and implemented the pending rollback reconciliation baseline:
  - `deploymentStatuses` now includes `rollback_pending`,
  - rollback execution writes `restore_in_flight` before provider restore and `rollback_pending` when the provider returns queued with a restored deploy id,
  - `SiteHostingPort` now exposes `getPublishedDeploy(...)` so reconciliation proves current published identity instead of only deploy readiness,
  - the rollback reconciler reads pending deployments, never calls `rollbackDeploy()`, and marks `rolled_back` only through the guarded success transaction when the intended provider deploy is currently published,
  - provider read failures and still-published target deployments stay pending,
  - third-deploy identity mismatches or malformed evidence move to manual reconciliation,
  - duplicate completion races are counted as `staleNoop` and do not overwrite terminal success evidence.
- Merged the pending rollback reconciler through PR #8 and accepted ADR 0014 for rollback trigger policy:
  - rollback after `rollback_recommended` remains explicit human/operator action for MVP,
  - automatic rollback is deferred as a per-project opt-in feature, default off,
  - future automatic rollback must require verified-good rollback sources, reproduced/debounced verification blockers, single-flight execution, loop prevention, `system_auto` audit, and customer/operator visibility,
  - Mastra/AI may explain or recommend rollback but must not enqueue rollback or call provider mutation ports.
- Merged ADR 0014 through PR #9 and completed `gsc-oauth-port-boundary`:
  - `SearchConsolePort` now owns `createAuthorizationRequest(...)` and the authorization request shape,
  - `GscService` depends on `SearchConsolePort | undefined` instead of the concrete `GoogleSearchConsoleAdapter`,
  - PKCE/state signing stays in the adapter while Redis nonce/code-verifier persistence stays in `GscOAuthStateStore`,
  - `exchangeCode(...)` now declares the PKCE `codeVerifier` requirement at the port boundary.
- Parked the remaining non-blocking review residuals in the lifecycle hardening backlog:

  - job-path rollback completion race audit noise is tracked separately from durable rollback truth,
