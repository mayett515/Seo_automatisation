# 2026-07-08

## Page Proposal UI Trigger And Status

- Added the operator-facing Page Proposal trigger in Opportunity Explorer:
  - selected opportunities can queue `POST /projects/:projectId/pages/proposals/runs`;
  - request and response bodies are parsed through the shared contracts;
  - rejected opportunities, already-created briefs, non-page-proposal recommendations, and active per-opportunity runs disable the action.
- Added Page Proposal run status to the frontend:
  - `AgentRunSummary` now exposes optional `subjectId`;
  - Opportunity Explorer reads `agent-runs?task=page_brief_draft`;
  - the selected opportunity panel shows the latest run for that opportunity;
  - the lower run area shows scout runs and page proposal runs separately.
- Kept the product boundary intact:
  - the UI queues durable work only through the API;
  - it does not call the model, worker, Mastra, or page persistence directly;
  - success still produces only draft page proposals and preview page versions through the worker.
- Switched Opportunity Explorer from route-ambiguous params to a typed route-owned `projectId` prop, matching the Pages screen pattern.
- Added an executable regression guard so the Page Proposal UI keeps parsing queue responses through contracts, calls the durable API endpoint, and reads status from `page_brief_draft` agent runs.
