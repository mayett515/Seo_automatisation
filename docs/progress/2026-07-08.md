## Page Version Approval Flow

- Added shared contracts for page-version review:
  - `ReviewPageVersionRequestSchema` accepts `approve` or `request_changes`;
  - requesting changes requires a decision note;
  - `PageVersionReviewResponseSchema` returns the updated page-version summary plus a durable approval audit record.
- Added `page:approve` as an explicit project permission for owner/admin/editor roles.
- Added `POST /projects/:projectId/pages/:pageVersionId/review`:
  - loads the page version through the project-scoped `pageVersions -> pageProposals` join;
  - parses and registry-validates the stored PageJson before review;
  - only reviews `preview` or `changes_requested` versions;
  - locks the page-version row and blocks `approve` while unresolved `approval_blocker` section notes exist;
  - sets `page_versions.status = approved` and `approvedAt` only for approvals;
  - writes `changes_requested` for request-changes decisions without approval evidence;
  - persists an `approvals` audit row tied to the actor, page version, decision, note, and timestamp.
- Extended the Pages preview surface:
  - loads page-version detail beside the sandboxed preview;
  - lists section notes through the existing notes query;
  - shows approval blocker state;
  - lets operators approve or request changes with visible pending/error/success feedback.
- Added integration coverage for:
  - successful approval and approval audit persistence;
  - unresolved `approval_blocker` notes blocking approval until resolved;
  - request-changes decisions not approving the page version;
  - request-changes note validation;
  - already-approved versions not being reviewed again.
- Updated ADR 0017, ADR 0019, the agent-first roadmap, the frontend/page-registry architecture doc, the Page Studio architecture note, backend foundation status, and the regression guard docs/script.
- Added a DB trigger preventing unresolved `approval_blocker` notes from being inserted or re-opened on non-reviewable page versions, while keeping ordinary notes allowed on approved versions.
- Added DB integration coverage for the approval/blocker race: an uncommitted `approval_blocker` insert holds the page-version lock, approval waits, then rejects after the blocker commits.

## Release Plan Creation From Approved Page Versions

- Tightened `POST /projects/:projectId/releases/plan` so release planning consumes approved page versions only:
  - the request contract requires at least one page version id;
  - page versions are loaded through the project-scoped `pageVersions -> pageProposals` join;
  - preview, changes-requested, release-candidate, released, superseded, missing, cross-project, or approval-evidence-missing versions are rejected;
  - successful planning creates a `draft` release plan and `pending` `create` release items only.
- Added a Pages preview action that can create a release plan only after the current page version is approved.
- Kept the lifecycle boundary intact:
  - release planning does not approve deploy;
  - release planning does not enqueue deploy;
  - release planning does not mutate providers;
  - release preflight, release approval, deploy, and verification remain separate deterministic steps.
- Added integration coverage for successful approved-version planning, preview-version rejection, non-approved immutable-version rejection, cross-project rejection, and absolute-route rejection.
- Added regression guards for the contract, API approval-evidence gate, integration coverage, and frontend API/contract usage.

Follow-up hardening:

- Tightened PageJson URL safety:
  - PageJson string safety now strips ASCII control characters before detecting dangerous URL schemes;
  - `Header.phoneHref` accepts only contact-safe `tel:` or `mailto:` links;
  - PageJson paths reject protocol-relative URLs and backslashes before they can reach rendered links or canonicals.
- Aligned rendering order with Page Studio composition order:
  - page-registry rendering, SEO fact extraction, and internal-link extraction now walk sections by `section.order`;
  - added registry tests proving render output follows `section.order`, not raw array position.
- Hardened Page Proposal persistence:
  - the success transaction no longer overwrites rejected opportunities with `brief_created`;
  - late opportunity lifecycle conflicts mark the run `qa_rejected:opportunity_lifecycle`.
- Added release-planning actor evidence:
  - `release_plans.created_by_user_id` records the persisted user who created the plan;
  - release-plan creation and release deploy approval reject missing/non-persisted actors instead of returning success-shaped side effects;
  - deploy approval no longer falls back to a scaffold user.
- Updated roadmap, ADR 0019, frontend architecture notes, lifecycle integration coverage, and executable regression guards for these hardening seams.
- Clarified rule routing after review:
  - rule 15 now routes explicitly for PageJson safety, approval/blocker serialization, Page Proposal lifecycle, and release-plan actor evidence;
  - TanStack Form remains the default for non-trivial persisted forms, while tiny one-field decision controls may use local state when contract parsing, mutation states, and error ownership are explicit.
