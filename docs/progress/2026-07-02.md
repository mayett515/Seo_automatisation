## Page Studio Direction

- Added [Page Studio Layout-Zone Editor](../architecture/page-studio-layout-zone-editor.md).
- Pinned the Page Studio product decision:
  - it is the "WordPress but easier" surface for local pages and subpages,
  - it is a constrained layout-zone editor, not a freeform drag-and-drop builder,
  - header/footer/hero/final CTA have realistic locked or limited movement,
  - body sections can move only inside allowed zones,
  - variant arrows, legal movement controls, text generation, media editing, notes, and approval all validate against the same page-registry rules.
- Updated the roadmap and frontend docs so Page Studio lives across slices 6-8: page proposal workflow, page registry/preview, and Page Studio notes/approval/versioning.

## Opportunity Scout Roadmap Refinement

- Updated [Agent-First MVP Roadmap](../architecture/agent-first-mvp-roadmap.md) after the read-only architecture review of the Big Eater Mastra/RAG/agentic UI findings.
- Replaced the older coarse slice numbering with the refined implementation sequence:
  - AI reasoning/opportunity boundary,
  - opportunity QA hardening,
  - `MockReasoningAdapter`,
  - Opportunity Scout worker vertical,
  - real reasoning adapter,
  - Opportunity Explorer plus manual evidence entry,
  - SERP/competitor snapshots,
  - Page Registry,
  - Page Proposal,
  - Page Studio,
  - release/report handoff,
  - RAG only when retrieval triggers become real.
- Added the manual evidence bridge decision:
  - `ranking_proof`, `serp_snapshot`, `field_evidence`, and `manual_note` are already contract vocabulary,
  - only evidence backed by project-owned rows/artifacts can pass resolution,
  - a small operator-entered ranking proof path in Opportunity Explorer makes `proven_win` reachable before automated SERP snapshots exist.
- Pinned the Opportunity Scout worker invariant in the AI boundary doc:
  - opportunities linked to a run may exist only when `agent_runs.status = "succeeded"`,
  - zero-brief success is legal,
  - retries and concurrent same-run delivery must not duplicate opportunity rows,
  - success writes opportunities and flips the run to `succeeded` in one transaction.
- Folded in the worker-plan review correction:
  - `failed -> running` is required for BullMQ retries after transient adapter failures,
  - `succeeded` is terminal and immutable,
  - `runId` should be both `agent_runs.id` and the BullMQ job id,
  - one `agent_runs` row spans all attempts while `job_runs` remains queue telemetry,
  - the worker should follow the website-import style: enqueuer creates the audit row, worker loads it and flips status.
