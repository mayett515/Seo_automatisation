  - `captured_at` for proof freshness policy later.

## Design Clarification

- DeepSeek can drive the future SERP/search workflow:
  - plan queries,
  - choose nearby Orte/service combinations,
  - call read-only SERP tools,
  - interpret the results.
- DeepSeek does not create ranking truth by itself. The evidence boundary is the stored `serp_snapshots` row or artifact.
- CAPTCHA/anti-bot is modeled as `captcha_blocked`, a stop signal rather than something to bypass.

## SERP Scout Worker Baseline

- Added `serp_scout` as a first-class job type and `serp-scout` as a worker/API queue.
- Added `MockSerpScoutAdapter` in `packages/adapters` so the SERP path can be exercised without live Google/search-provider calls.
- Wired `handleSerpScoutJob` in `apps/worker`:
  - parses typed job data,
  - calls `SerpScoutPort.search(...)`,
  - persists captured snapshots into `serp_snapshots`,
  - records failed snapshots with normalized engine errors,
  - retries transient provider failures,
  - treats not-configured, captcha, and policy failures as terminal.
- Added API enqueue support:
  - `POST /projects/:id/serp-scout/runs`,
  - guarded by `opportunity:run`,
  - `jobId = snapshotId`,
  - dry-run semantics when DB/Redis is absent,
  - `job_runs` audit via the existing queue producer.
- Added focused worker and API tests for parsing, route wiring, failure classification, mock capture, queue audit, and dry-run behavior.

## OpenCode Go Model Routing Research

- Recorded the Big Eater model-routing finding:
  - the current OpenCode Go adapter supports the `/chat/completions` family,
  - `deepseek-v4-flash`, `deepseek-v4-pro`, `glm-5.2`, `kimi-k2.7-code`, and `mimo-v2.5*` are available through that lane,
  - `minimax-m3` and the `qwen3.7*` family need a later `/messages` adapter.
- Adjusted docs so `deepseek-v4-flash` is the first practical real smoke/default for cheap repeated worker runs.
- Kept `deepseek-v4-pro` as the harder reasoning fallback.
- Kept `glm-5.2` as a strong candidate for page/layout/content/report-quality work instead of a product contract.
- Model routing remains runtime adapter policy; contracts, DB, UI, QA, and evidence truth stay model-neutral.

## SERP Provider Strategy Research

- Recorded the provider-strategy conclusion from Big Eater:
  - F.2 is an adapter/proof-policy decision, not an LLM web-search decision.
  - If dedicated SEO APIs are acceptable, Serper is the cheap POC candidate, DataForSEO is the strongest proof-grade production candidate, and SerpApi is the mature fallback/reference.
  - If dedicated SEO APIs are not acceptable, manual `ranking_proofs` remain customer-safe proof while Brave/Tavily/model search stays discovery/context only.
- Strengthened the F.1 guard:
  - not every `serp_snapshot` can support `proven_win`,
  - QA must enforce status, freshness, exact result attribution, provider/searchEngine, and result type before a snapshot can be customer-safe proof.
