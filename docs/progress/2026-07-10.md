# 2026-07-10

## Bounded Stale-Work Recovery Scanner

- Added the first procedural consumer of ADR 0018's pure recovery classifier.
- The worker host now scans stale Page Proposal `agent_runs` and running `release_verifications` on the existing lifecycle interval.
- Added first-class durable recovery evidence:
  - `recovery_count` bounds automatic attempts;
  - `last_recovery_at` records the latest claim;
  - partial indexes support stale active-row scans.
- Recovery combines three sources without confusing their ownership:
  - `agent_runs` / `release_verifications` remain product truth;
  - `job_runs` remains queue/audit telemetry;
  - BullMQ remains transport state.
- Safe missing/failed transport is re-enqueued with the same deterministic run or verification id only after a guarded Postgres claim.
- Competing scanners cannot duplicate recovery because the claim checks active status, stale time, and the observed recovery count.
- Unknown BullMQ state is a conservative no-op.
- Completed transport without terminal Page Proposal truth becomes `work_transport_inconsistent` instead of being replayed.
- Bounded Page Proposal exhaustion becomes `work_recovery_exhausted`.
- Bounded release-verification exhaustion becomes `execution_failed` with a warning-level `verification_recovery_check`; it does not claim observed bad page health.
- Recovery attempts create system-owned `job_runs` audit rows with `trigger_source = work_recovery`, and failed recovery enqueue remains visible there for the next bounded scan.
- The scanner registers only `page-generation` and `release-verification`. Deploy and rollback provider mutations remain excluded and continue through their existing reconcilers/manual-reconciliation boundaries.
- Added unit and DB-backed integration coverage for transport mapping, deterministic re-enqueue, durable audit, exhaustion, completed-transport inconsistency, queue failure, and two scanners racing one stale run.
- Ran the worker integration suite against the documented local disposable Postgres runtime: 75 tests passed, including all six bounded-recovery cases.
- Updated ADR 0018, the lifecycle hardening backlog, roadmap, backend status, project regression rules, environment example, and executable guards.
