# 2026-06-30 Progress

## Scope

Deploy provider-operation hardening after Netlify reconciliation pattern-mining and follow-up architecture reviews, plus the first real HTTP post-deploy verifier baseline.

## Completed

- Promoted `provider_operation_status` to a contract-owned enum with `manual_reconciliation_required`.
- Split `SiteHostingPort` into phased provider operations:
  - `beginDeploy`,
  - `uploadDeployFiles`,
  - existing `getDeploy`,
  - composed `createDeploy` retained for convenience.
- Changed the Netlify adapter to:
  - send a traceable deploy title,
  - create the async digest deploy before upload,
  - poll for required file digests,
  - return an opaque resume token,
  - upload files from the supplied resume token.
- Changed the deploy worker to persist `providerDeployId` and upload resume evidence before file upload.
- Added upload-resume behavior for recorded provider deploys so a retry after a mid-upload crash checks provider state and re-uploads from persisted evidence when local upload completion is not recorded.
- Changed `in_flight` without a recorded provider deploy id from ordinary `failed` to `manual_reconciliation_required`.
- Added worker and repository guards so manual reconciliation rows are not overwritten by `startDeployment`, `markProviderMutationInFlight`, provider success/pending writes, or failure handling.
- Added ADR 0009 to record the accepted decisions and rejected ideas:
  - no automatic recreate from unknown provider state,
  - no advisory locks across provider I/O,
  - no automatic provider-resource deletion compensation,
  - no recovery-point/provider-operation table until the model needs it.
- Replaced the synthetic release `verify()` response with a `VerificationPort` backed by `HttpReleaseVerificationAdapter`.
- The verifier now checks intended live routes derived from deployment live URL + release item routes:
  - HTTP success,
  - noindex robots directives,
  - canonical URL/trailing slash match,
  - JSON-LD parseability,
  - sitemap inclusion,
  - tracking marker presence when tracking is configured.
- Release verification now persists `release_verifications` and `release_verification_checks`, then updates deployment verification status and deployment health status from the observed checks.
- Follow-up verification fixes aligned the renderer and verifier contract so rendered pages can emit approved canonical and JSON-LD fields, prefer stable production URLs over deploy permalinks, avoid a committed `verifying/running` pre-state in the inline API path, update the release plan status from verification truth, and match sitemap `<loc>` entries exactly.
- Started Foundation Milestone 3 lifecycle integration coverage:
  - added `test:integration` scripts,
  - added a disposable Postgres migration/reset harness guarded by `TEST_DATABASE_URL`,
  - added the first `ReleasesService.verify()` API/DB integration spec covering healthy verification, rollback recommendation, verifier failure evidence, release-plan status projection, and cross-project rejection.
- Extended lifecycle integration coverage with real DB tests for:
  - deploy worker provider success, manual reconciliation guards, in-flight escalation, upload resume, and reconciler manual-row filtering,
  - queue/job audit dry-run truth, duplicate active-job coalescing, terminal re-enqueue archival, and queue-add failure evidence,
  - tracking ingestion valid-key persistence, revoked-key rejection, origin rejection, cross-project key rejection, and `lastUsedAt` coalescing.
- Review follow-up tightened Milestone 3 proof:
  - `verify()` now persists the scoped route `releasePlanId`, not adapter-returned identity,
  - release verification integration now covers adapter identity drift and cross-plan deployment id rejection,
  - deploy worker integration now covers the `markFailed` manual-state guard and pending provider reconciliation,
  - CI now has a PostgreSQL-backed `test:integration` job.
- Started and wired Foundation Milestone 4 rollback execution:
  - added `RollbackJobDataSchema`, `ExecuteRollbackRequestSchema`, the `rollback` queue name, and the `rollback:execute` project permission,
  - added `POST /projects/:projectId/releases/:releasePlanId/rollback/execute` to authorize, scope, validate, queue, and audit rollback execution without mutating success state in the API,
  - added a deterministic rollback worker that reloads release, pinned target deployment, rollback point, and hosting evidence from Postgres before calling the provider-neutral hosting rollback operation,
  - added Netlify rollback support through the provider restore endpoint inside the adapter,
  - made `releasePlans.status = "rolled_back"` only after provider rollback completion is persisted by the worker, with transaction guards against stale deployment or release-plan state,
  - recorded provider-pending rollback results without retrying the restore mutation in the same worker,
  - added API integration coverage for rollback queueing, rollback-point scoping, rollback-ready release-plan gating, missing target provider evidence, and missing rollback-point provider evidence,
  - added worker integration coverage for completed rollback, terminal provider rollback failure, provider-pending rollback, stale release-plan/deployment state, pinned target deployment updates, terminal not-configured results, and missing rollback evidence,
  - hardened rollback execution evidence so pending/failed attempts store provider-neutral `providerResultStatus`, `providerDeployId`, and `liveUrl`, and successful rollback records the bad provider deploy id that was rolled back from,
  - added ADR 0011 to record the rollback restore execution lifecycle.
- Recorded the accepted DeepSeek/Opus lifecycle-truth hardening notes in `docs/architecture/lifecycle-truth-hardening-backlog.md` so the next patch can address verifier execution-error semantics and verification target URL safety without mixing that work into the rollback execution commit.
- Wired rollback point preparation into release preflight:
  - preflight now snapshots the latest provider-backed prior deployment into `rollback_points` for the new release before evaluating `rollback_point_ready`,
  - rollback point rows without provider deploy evidence no longer satisfy API preflight or the deploy worker's final rollback-evidence check,
  - API integration coverage now proves automatic rollback point preparation and fail-closed behavior for placeholder rollback rows.
- Landed verifier truth hardening from the DeepSeek/Opus review notes:
  - added `execution_failed` to `release_verification_status` so verifier infrastructure failures no longer masquerade as observed live-page failures,
  - `verify()` now preserves deployment/release health projection when the verifier crashes and persists skipped execution-error evidence,
  - release plan creation and verification now reject absolute or protocol-relative target routes before verifier fetches can leave the deployment host,
  - API integration coverage now proves execution-error persistence, pre-verifier absolute-route rejection, and release-plan creation rejection for absolute page proposal routes.

`test:integration` currently skips unless `TEST_DATABASE_URL` is configured. The configured database must be disposable; the harness resets the `public` schema and refuses to run unless the database name contains `test` or `integration` or `LOCALSEO_ALLOW_TEST_DB_RESET=true` is set.

PostgreSQL 17 is now available as a user-space runtime under `%USERPROFILE%\.localseo-tools`, and the configured integration command passes against `local_seo_test`. Integration coverage details and local database commands are documented in `docs/testing/lifecycle-integration-coverage.md`.
