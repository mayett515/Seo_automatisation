# 2026-07-03 Progress

## Opportunity Scout Worker Baseline

- Implemented the first Opportunity Scout worker vertical behind the existing AI reasoning boundary:
  - added the `opportunity_scout` job contract and `opportunity-scout` queue name,
  - added `MockReasoningAdapter` as the deterministic first `AiReasoningPort` implementation,
  - added a stable evidence-packet builder and prompt for the scout task,
  - wired the worker route for opportunity scout jobs,
  - added a Drizzle-backed repository that loads project-scoped website import, GSC, tracking, existing-route, and open-opportunity evidence,
  - writes the evidence packet through `ObjectStoragePort` as `agent_runs.input_ref`,
  - parses model output through `OpportunityScoutOutputSchema`,
  - runs deterministic QA/scoring before persistence,
  - inserts opportunities and marks the run `succeeded` in one transaction.
- Preserved the run-state decisions from the roadmap review:
  - `queued -> running` starts a run,
  - `failed -> running` supports BullMQ retries after transient adapter failures,
  - provider failures are retryable worker errors,
  - schema/QA/evidence/config failures are terminal worker errors,
  - `succeeded` is terminal and replays as a no-op,
  - zero-brief success is valid and persists no opportunity rows.
- Added coverage for the worker baseline:
  - mock adapter unit tests,
  - pure evidence-packet ordering test,
  - worker unit tests for success, adapter failure, schema mismatch, QA rejection, and route/error classification,
  - DB-backed integration tests for success persistence, zero-brief success, provider-failure retry to success, no duplicate replay, and cross-project evidence rejection.

## Opportunity Scout API Enqueue

- Added the operator-facing API entrypoint for the worker baseline:
  - `POST /projects/:id/opportunity-scout/runs`,
  - request body accepts only `maxBriefs`,
  - route uses the existing project guard chain and the new `opportunity:run` permission,
  - owner/admin/editor can run the scout; viewers cannot.
- Preserved the worker/run invariant at the enqueue boundary:
  - when DB and Redis queue are configured, the API creates `agent_runs` as `queued`,
  - BullMQ `jobId` is exactly the `runId`,
  - queue job data carries `projectId`, `runId`, optional `maxBriefs`, actor metadata, and the `jobRunId` generated by queue audit,
  - queue add failures mark the queued `agent_runs` row `failed` with queue diagnostics.
- Kept local infrastructure behavior explicit:
  - when DB or the opportunity queue is not configured, the endpoint returns a `dry_run` queue response,
  - dry-run mode does not create an `agent_runs` row because no worker can consume it.
- Added integration coverage for:
  - queued `agent_runs` creation,
  - `jobId = runId`,
  - job audit row creation,
  - explicit dry-run without agent rows,
  - queue-add failure cleanup.

## OpenCode Go Reasoning Adapter Baseline

- Added the first real `AiReasoningPort` provider adapter:
  - `OpenCodeGoReasoningAdapter` lives in `packages/adapters`,
  - calls the OpenCode Go OpenAI-compatible chat-completions endpoint,
  - defaults to model `glm-5.2`,
  - returns untrusted `outputJson` only after the provider response and assistant content are parseable JSON.
- Kept the worker safe by default:
  - `AI_REASONING_PROVIDER` defaults to `mock`,
  - `AI_REASONING_PROVIDER=opencode_go` requires `AI_REASONING_OPENCODE_GO_API_KEY`,
  - `AI_REASONING_OPENCODE_GO_ENDPOINT` can override the default endpoint,
  - `AI_REASONING_TIMEOUT_MS` is passed into `runStructured`.
- Mapped provider failures into the existing reasoning failure taxonomy:
  - timeout/abort -> `provider_timeout`,
  - missing required provider config -> `provider_not_configured`,
  - HTTP 429/503 -> `provider_overloaded`,
  - other non-2xx provider responses -> `provider_error`,
  - invalid completion JSON or non-JSON assistant content -> `output_not_json`.
- Added focused coverage for:
  - adapter success request shape and JSON parsing,
  - provider error body redaction,
  - overload, invalid JSON, non-JSON content, and timeout mapping,
  - config defaults and explicit OpenCode Go config,
  - worker composition root adapter selection,
  - configured reasoning timeout reaching the scout worker call.
- Still deferred:
  - Mastra internals behind the same port,
  - OpenCode Go `/messages` endpoint support for MiniMax/Qwen-family models,
  - real-provider smoke run with a project API key,
  - Opportunity Explorer and manual ranking evidence bridge.

## Manual Ranking Proof Bridge

- Added the first backing source for `ranking_proof` evidence before automated SERP snapshots:
  - new `ranking_proofs` table stores project-owned manual SERP proof,
  - API endpoint shape lives under `projects/:projectId/ranking-proofs`,
  - `opportunity:evidence` permission lets owner/admin/editor create proof while preserving project guards,
  - worker evidence loading now includes ranking proof rows in both the model input packet and `resolvableEvidence`.
- This makes `proven_win` reachable without weakening the proof gate:
  - the model still must cite a `ranking_proof` row in the brief's own evidence,
  - the claimed rank, query, and page URL must match the cited `ranking_proofs` row,
  - cross-project proof remains rejected by evidence resolution,
  - GSC remains internal radar only.
- Still deferred:
  - Opportunity Explorer UI form and evidence panel,
  - automated SERP/competitor snapshots and proof freshness policy.
